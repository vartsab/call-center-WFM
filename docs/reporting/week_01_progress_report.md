# Щотижневий звіт про прогрес. Тиждень 1

## Проєкт

**Call Center Analytics and Workforce Optimization for City Service Operations**

Дата підготовки звіту: 11.05.2026  
Звітний період: 1-й тиждень активного етапу дипломного проєктування

## 1. Виконані завдання, досягнення та зміни у проєкті

Протягом першого тижня було сформовано та деталізовано концепцію магістерського інженерного проєкту. Проєкт визначено як end-to-end аналітичну систему для контакт-центру міських сервісів, яка поєднує інженерію даних, базу даних MS SQL Server, інтерактивний дашборд, прогнозування навантаження, розрахунок потреби у персоналі та оптимізацію графіків роботи.

Основні виконані завдання:

- визначено тип дипломної роботи: інженерний проєкт із компонентом Data Science;
- сформульовано проблему: контакт-центрам складно планувати покриття персоналом через змінне навантаження, сезонність, пікові періоди та вимоги до SLA;
- визначено цільових користувачів: менеджер контакт-центру, аналітик Workforce Management, аналітик міських сервісів;
- описано основну цінність системи: перетворення історичних даних про звернення на операційні рішення щодо прогнозу, staffing та scheduling;
- створено структуру репозиторію для коду, SQL-скриптів, документації, даних і звітності;
- підготовлено початкові документи проєкту: project brief, architecture, decision log, data dictionary, capstone report outline, accelerated execution plan;
- обрано початкову архітектуру системи: public 311 data → synthetic call metadata → SQL Server star schema → SQL views → Streamlit dashboard → forecasting → Erlang C → schedule optimization;
- визначено підхід до даних: гібридна стратегія, де реальні публічні 311-дані використовуються як seed для попиту, а відсутні операційні поля контакт-центру генеруються синтетично;
- зафіксовано рішення використовувати 30-хвилинні інтервали як основну операційну гранулярність для прогнозування, розрахунку FTE та планування змін.

## 2. Концепція, проблема та цілі проєкту

Проєкт спрямований на вирішення задачі планування роботи контакт-центру міських сервісів. Попит на звернення змінюється залежно від часу доби, дня тижня, сезону, типу звернення, району міста та зовнішніх подій. Через це менеджеру контакт-центру потрібно не лише бачити історичну статистику, а й прогнозувати навантаження та визначати, скільки операторів потрібно в кожному 30-хвилинному інтервалі.

Мета проєкту — створити відтворювану аналітичну систему, яка:

- збирає або формує реалістичний набір даних контакт-центру;
- моделює дані в MS SQL Server у вигляді star schema;
- надає SQL views для аналітики та дашбордів;
- будує дашборд у Streamlit;
- прогнозує кількість звернень у 30-хвилинних інтервалах;
- розраховує необхідне staffing-покриття через Erlang C;
- формує оптимізований графік змін за допомогою методів операційного дослідження.

Очікуваний результат — MVP системи, яка демонструє повний шлях від даних до управлінського рішення.

## 3. Початкова версія MVP та методології

На першому тижні було визначено модульну структуру MVP:

1. Data acquisition: завантаження реальних публічних 311-даних.
2. Synthetic generation: створення операційних полів, яких немає у відкритому наборі даних.
3. SQL Server warehouse: побудова таблиць фактів і вимірів.
4. SQL analytics layer: створення views для KPI, дашбордів і ML-пайплайна.
5. Forecasting: прогнозування кількості звернень у 30-хвилинних інтервалах.
6. Erlang C staffing: розрахунок необхідної кількості операторів.
7. Shift scheduling: оптимізація графіків роботи команди.
8. Streamlit dashboard: інтерактивне представлення результатів.

На цьому етапі MVP був визначений на рівні архітектури, структури репозиторію, даних, майбутньої бази даних і ключових методів реалізації.

## 4. Ризики та виклики

| Ризик або виклик | Чому це важливо | План реагування |
| --- | --- | --- |
| Публічні 311-дані не містять реальних call-center полів. | У відкритих даних зазвичай немає talk time, wait time, ACW, agent ID, SLA та abandonment. | Використати реальні 311-дані як seed для попиту, а операційні поля створити синтетично з прозорими припущеннями. |
| Надто широкий scope проєкту. | Проєкт включає ETL, SQL, dashboard, ML, Erlang C та оптимізацію. | Рухатися MVP-підходом: спочатку робочий мінімум для кожного модуля, потім покращення. |
| Синтетичні дані можуть бути поставлені під сумнів. | Від якості припущень залежить академічна переконливість проєкту. | Документувати всі припущення, random seed, розподіли та обмеження. |
| Налаштування SQL Server може затримати реалізацію. | Без бази даних складніше будувати views і dashboard. | Паралельно з SQL тримати CSV-проміжні файли, щоб ML і dashboard могли рухатися вперед. |
| Прогнозування низьких інтервалів може бути нестабільним. | Помилки прогнозу впливають на staffing. | Порівняти модель із baseline та обережно використовувати метрики для малих значень. |
